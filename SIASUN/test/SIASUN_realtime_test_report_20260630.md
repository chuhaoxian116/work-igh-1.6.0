## 2. 测试场景

目标设备共有 8 个逻辑 CPU、4 个物理核心，超线程关系为
`0/4`、`1/5`、`2/6`、`3/7`。实时通信应用固定在 CPU 2。

| 编号 | 压力类型 | 压力 CPU | 压力配置 | 与实时应用的关系 |
| --- | --- | --- | --- | --- |
| T01 | 空载基线 | - | 不启动 stress-ng | 仅运行系统基础任务和 EtherCAT 通信 |
| T03 | CPU | 1,5 | 2 个 CPU worker，100% | 占满另一个物理核心 |
| T04A | CPU | 2 | 1 个 CPU worker，100% | 与实时应用位于同一逻辑 CPU |
| T04B | CPU | 6 | 1 个 CPU worker，100% | 与 CPU 2 共享同一物理核心 |
| T05 | 内存 | 1,5 | 1 个 VM worker，8 GiB | 内存读写压力位于另一物理核心 |
| T07 | CPU+内存 | 1,5 | 2 个 CPU worker 100%，1 个 VM worker 8 GiB | 3 个 worker 竞争 2 个逻辑 CPU |

压力进程为普通低优先级任务，实时应用为 SCHED_FIFO 99。因此在
T04A 中，实时应用仍可抢占同一 CPU 上的压力线程。

## 4. 结果分析

### 4.1 通信可靠性

空载基线及五种压力条件下，EtherCAT 通信完整性结果一致：

- 7 个从站始终在线并进入 OP。
- Domain 工作计数始终为期望值 21。
- 未出现不完整周期、无数据周期和 Domain 读取失败。
- 未出现大于 1.5 ms 的严重超周期。
- DC 样本全部有效。
- Servo 6 持续运动，测试结束时无驱动错误。

因此，在本轮每组 30 分钟的测试范围内，当前系统通过了空载、CPU、
内存及混合压力下的基本实时通信稳定性测试。

### 4.2 空载基线

T01 未启动 stress-ng，平均绝对抖动为 1.978 us，DC 平均偏差为
0.710 us。其最大绝对抖动为 78.862 us，是本轮六组测试中的最大值，
但仍远低于 1.5 ms 的严重超周期门槛，且没有造成 WC 或 DC 样本异常。

T01 的应用 CPU 占用为 2.897%，同样是六组最高。这说明当前空载条件下
CPU 可能更频繁地降频或进入深度空闲状态，使相同通信工作消耗更多执行
时间，并增加偶发唤醒延迟。当前日志未记录实时 CPU 频率和 C-state，
所以该原因仍需进一步验证。

### 4.3 T04A 为何表现最好

T04A 在 CPU 2 上运行一个普通压力线程，同时实时通信线程也固定在
CPU 2。由于通信线程使用 SCHED_FIFO 99，它可以直接抢占压力线程，
不会像两个同优先级实时线程那样发生不可控竞争。

本组平均绝对抖动为 0.468 us，DC 平均偏差为 0.313 us，均为六组最小。
相对 T01，平均绝对抖动降低约 76.3%，DC 平均偏差降低约 55.9%。
一种合理解释是持续负载使 CPU 2 保持活跃，减少了降频、深度空闲状态
退出等造成的唤醒延迟。该解释目前属于推测，还没有 CPU 频率和 C-state
数据作为直接证据。

T04A 的结果不能直接推出“生产环境应当主动拉满 CPU 2”。压力线程仍会
消耗功耗和温度裕量，并可能影响同核的其他任务。更合理的后续验证方式是
固定 performance governor、限制深度 C-state 后重新测试，观察是否能在
不运行压力线程的情况下得到同等效果。

### 4.4 T04B 的超线程影响

CPU 2 和 CPU 6 属于同一个物理核心。T04B 把压力放在 CPU 6，会与实时
线程共享执行单元、缓存和核心功耗预算。

该组平均抖动 1.714 us，较 T01 降低约 13.3%，优于 T03，但弱于
T04A；DC 最大偏差 8.191 us 为本轮最小。由于每个场景仅运行一次，而
最大值对偶发事件敏感，暂不能仅根据该最大值认定超线程同核配置最优。

### 4.5 T03 与 T07

T03 将另一个物理核心的两个超线程完全拉满，但其平均抖动 7.370 us、
DC 平均偏差 1.762 us，均为六组最大；应用自身 CPU 时间也明显高于多数
组。这说明本轮差异不只取决于“压力是否靠近 CPU 2”，还可能受到 CPU
频率、空闲状态、IRQ 活动、启动阶段系统状态或测试顺序的影响。

T07 同时施加 CPU 与 8 GiB 内存压力。其最大周期抖动 67.133 us 为五个
压力场景中最大，但仍低于 T01 空载基线的 78.862 us，且未发生严重超
周期、工作计数异常或无效 DC。测试结束时 CPU PSI 约为 48%，说明指定
CPU 集合内存在真实的运行队列竞争。

### 4.6 内存压力的边界

T05 和 T07 的 8 GiB VM 压力均未触发 swap、OOM 或 memory PSI。它们
可以证明通信在 8 GiB 内存读写负载下保持稳定，但不能证明系统在内存
回收、交换或接近 OOM 的强压力下仍有相同性能。

## 5. 初步结论

1. 六组测试的通信成功率均为 100%，工作计数固定为 21/21，7 个从站
   全部保持 OP，当前系统的基本通信稳定性良好。
2. 六组平均周期均接近 1 ms，最大周期为 1078.862 us，没有出现大于
   1.5 ms 的严重超周期。
3. 所有 DC 样本均有效，平均偏差范围为 0.313 至 1.762 us，最大偏差
   不超过 32.767 us。
4. T01 建立了无压力基线。T04A 的平均周期抖动和 DC 平均偏差明显优于
   该基线，T04B 的 DC 最大偏差最小。这些结果可以作为后续 CPU 电源
   管理和亲和性实验的重点，但尚不足以确定最终 CPU 配置。
5. T07 是本轮压力最复杂的场景，最大抖动有所增加，但未影响 PDO 完整性、
   DC 有效性和伺服运动。
6. 8 GiB 内存测试没有形成内存回收压力，当前只能评价内存带宽及容量占用
   条件下的表现。

## 6. 当前报告限制

- 每种场景只有一次测试，无法排除测试顺序、温度和偶发系统活动的影响。
- 当前统计只有平均值和最大值，缺少 P99、P99.9 和抖动直方图。
- 日志未记录实时测试期间的 CPU 实时频率、governor 和 C-state。
- 日志未完整记录 `EtherCAT-OP` 内核线程及网卡 IRQ 的实时调度策略和
  CPU 亲和性，因此本报告主要评价应用侧配置与最终通信结果。
- 普通网卡 IRQ 统计不等同于 IgH 原生驱动轮询线程的完整执行情况。

## 7. 建议的下一轮测试

1. 增加无压力基线测试，并保持与现有测试相同的 30 分钟时长。
2. 对 T03、T04A、T04B 和 T07 各重复至少 3 次，交错执行测试顺序。
3. 对比默认电源管理与 performance governor，并记录实时频率和
   C-state 计数。
4. 增加周期抖动和 DC 偏差的 P99、P99.9、P99.99 统计。
5. 分别记录应用线程、`EtherCAT-OP`、网卡 IRQ 的调度策略、优先级、
   CPU 亲和性和运行 CPU。
6. 在确认系统安全余量后，再进行更高内存占用测试；若目标是验证内存
   回收影响，应同时记录 memory PSI、swap 和主缺页。
7. 最终配置至少进行一次 8 至 24 小时的连续运动老化测试。

## 8. 数据来源

每组目录中的以下文件用于本报告：

- `communication.log`：周期、抖动、DC、Domain、主站、从站及伺服状态。
- `communication.stderr.log`：GNU time 进程资源统计。
- `environment.log`：内核、CPU、IRQ、压力配置及测试前后系统状态。
- `stress.log`：五个压力场景中的 stress-ng 任务及运行统计；T01 不启动
  stress-ng，因此没有该文件。
- `status.log`：测试脚本退出状态。

本报告为首版阶段性报告，结论仅适用于上述设备、内核、主站配置和测试
数据范围。
